# Live Agentic Workflow Upgrade Report

## Architecture Update

We have successfully upgraded the `agent_orchestrator.py` from deterministic mock fixtures to a **live LLM execution engine**. 

### New Components Introduced:
1. `agent_state.py`: Introduced the `AgentWorkflowState` model. This acts as the shared blackboard across all 12 agents, allowing them to accumulate extracted claims, verified facts, source metadata, and structured output.
2. `agent_prompt_templates.py`: Added explicit instructional prompts for all 12 agent specializations, enforcing JSON-only output and strict schema compliance.
3. `agent_output_validator.py`: Added `AgentOutputValidator` to parse LLM responses safely, strip out markdown fences (` ```json `), and validate the existence of critical keys per agent role.
4. `live_agent_executor.py`: Implemented `LiveAgentExecutor` which handles the agent loop. It invokes the AI Provider router, parses latency, handles fallback validation, and generates unified trace steps.

### Backend Routing

The backend now uses `os.environ["APP_MODE"]` and `os.environ["ENABLE_REAL_LLM"]` to dynamically switch between `mock` and `real` paths inside `run_full_workflow()`. 

When in `real` mode, the orchestrator triggers the `live_agent_executor`. Any agent that fails JSON validation, or any LLM failure, is gracefully routed to its deterministic fixture via fallback.

### Frontend Updates

- **Real-time Client Wrapping**: Converted the `AgentWorkflowPage` into a client wrapper (`AgentWorkflowClientWrapper.tsx`) to support a "Run Workflow" action and dynamic UI state toggles, effectively simulating the real-time polling logic.
- **Provider Metadata Badges**: Agent Trace UI now shows `provider_used` (e.g. Gemini, Mock) and `latency_ms` for each step. Fallback states are clearly flagged with an amber "Fallback" badge.
- **Settings UI**: The settings dashboard now correctly identifies the active LLM routing mode and default provider distribution, while displaying a testing action.
- **Memo Trace**: The `deal/[id]/memo` UI now indicates whether the memo was generated by the Memo Writer Agent in Mock or Live mode.

## Next Steps

2. Connect `AgentWorkflowClientWrapper` to a true polling mechanism (`setInterval`) if we shift the backend endpoints to asynchronous Celery/RQ workers in the future.
